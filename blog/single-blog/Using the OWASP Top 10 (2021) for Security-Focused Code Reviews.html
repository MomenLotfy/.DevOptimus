How DevOps is Transforming Modern Development

Published on July 17, 2025 • DevOps

Ensuring application security at the code-review stage is a pivotal step in catching vulnerabilities before they reach production. The OWASP Top 10 (2021)—a globally recognized set of the most critical web application risks—serves as an ideal framework for guiding security-focused code reviews. Below, we explore how each vulnerability category can be applied during review, with tips to identify and address issues early.

1. Broken Access Control (A01:2021)

Broken access control persists as the top risk identified in OWASP 2021, where attackers can manipulate APIs or endpoints to gain unauthorized privileges or access sensitive data. During code reviews, look for direct object references (e.g., using user IDs in URLs) and ensure authorization checks are applied consistently—even for admin-level actions.

Reviewers should verify that business logic enforces role-based constraints, avoid hardcoded permission schemes, and ensure every action is subject to a security gate. Tests that directly exercise permission boundaries—like accessing another user’s data—should be part of the PR checklist to validate runtime behavior.

2. Cryptographic Failures (A02:2021)

Cryptographic failures often come from the misuse of weak or outdated encryption, insecure random number usage, or storing secrets in plaintext. In a code review, validate that strong, industry-standard algorithms (e.g., AES-256, RSA-2048) are being used, and that secrets are retrieved from vaults or environment variables—not source files.

Reviewers must also check that HTTPS/TLS is enforced for sensitive data transmission, keys are rotated properly, and no deprecated methods (e.g., MD5, SHA1) remain in use. Include a checklist item to ensure all external dependencies supply secure defaults.

3. Injection (A03:2021)

Injection flaws—SQL, NoSQL, command-line, or even LDAP—continue to plague systems due to unsanitized inputs. During review, audit every instance where user-controlled input is parsed into database queries or system calls. Prefer parameterized queries, ORM tools, or safe query APIs over string concatenation.

Static analysis tools integrated in CI can flag some patterns, but reviewers should manually inspect complex query assembly code. Encourage use of whitelisting, escaping, and strong validation on inputs—particularly those used in shell commands, eval calls, or dynamic queries.

4. Insecure Design (A04:2021)

This new category emphasizes architecturally insecure patterns—such as missing threat modelling, unsafe default behaviors, or poor design assumptions. In code reviews, reviewers should question design decisions: is data flow properly segmented? Are fallback behaviors safe? Has threat modelling been referenced?

Ensure that upstream design documents (e.g., diagrams, threat models) are present and accurately reflected in the code. Look for anti-patterns like default-allow behavior, global state usage, or centralised secrets loading. Code reviews should cross-reference design intentions with real implementation.

5. Security Misconfiguration (A05:2021)

Misconfiguration—from verbose error messages, open CORS APIs, or incorrect HTTP headers—often leads to simple yet impactful exposures. Security review should include config files: ensure debug flags are off in production, default credentials are removed, and relevant headers (HSTS, CSP) are set.

Inspect YAML, JSON, and env files for missing security flags. Validate that sensitive configuration is not hardcoded—use deployment-specific overrides. Confirm framework defaults have been hardened and insecure shortcuts (e.g., disabling SSL validation) are flagged in PR reviews.

6. Vulnerable and Outdated Components (A06:2021)

Outdated dependencies or frameworks pose significant risks. Ask whether the PR updates dependencies, and confirm that no components with known CVEs are being introduced. Reviewers should use automated SCA tools—like Snyk or OWASP Dependency‑Check—and manually confirm alerts before merging.

Audit transitive dependencies and ensure CVE severity is communicated. Even if an older version isn't deprecated, assess if a safer upgrade is warranted. Require changelog review, and avoid skipping updates unless a thorough risk evaluation is documented.

7. Identification and Authentication Failures (A07:2021)

Authentication weaknesses—such as weak password policies, missing MFA, or credential leaks—are rife. In code reviews, validate that password hashing uses bcrypt/Argon2, timeouts and account lockouts are in place, and token lifecycles are enforced.

Check email verification flows, password-reset endpoints, and secure cookie flags. Look for potential session fixation or lack of re-auth on sensitive actions. Also ensure MFA isn't bypassed via hidden endpoints or test flags.

8. Software and Data Integrity Failures (A08:2021)

This category covers unexpected code or data tampering—such as bypassing CI/CD integrity or allowing unverified plugin downloads. In review, check that artifacts are signed or validated, SBOMs are published, and import origins are verified through checksum or signature.

Review the build process to ensure it incorporates SLSA-level validations. Verify that the codebase rejects unverified external scripts or remote modules. Ensure any dynamic code fetching includes integrity validation (e.g., Subresource Integrity headers).

9. Security Logging & Monitoring Failures (A09:2021)

Without proper logging, detecting and reacting to intrusions is near impossible. Code reviewers should identify places where sensitive actions occur—authentication failures, permission checks—and confirm logging is enabled without leaking PII in logs.

Verify log levels, structured logs, and centralized sinks are used. Look for missing logs around data access or privilege escalations, and ensure logs include context, timestamps, and user IDs. Also confirm no sensitive values like passwords are logged.

10. Server-Side Request Forgery (SSRF) (A10:2021)

SSRF vulnerabilities let attackers trick the server into making requests on their behalf, often to internal services. Review code loading remote URLs based on user inputs—validate against whitelist hosts/IPs and reject private or loopback addresses.

For file loaders, image fetchers, or API proxies, confirm allowlists, URL normalisation, timeouts, and network segmentation. Include PR checks to validate that no open proxy logic is merged accidentally.

“Using OWASP Top 10 as a baseline for code reviews turns PR checks from guesswork into security-focused vetting—catching issues before deployment.”

By embedding each OWASP category into your code-review checklist—and enhancing reviews with static analysis, secure design documentation, and PR verification—you shift security left and dramatically reduce risk. It transforms code reviews from a cursory glance into a robust, preemptive line of defense.
